The application may be\n doing too much work on its main thread.\xe2\x80\x9d So what does it actually\n means, why should you be concerned and how to solve it.What this means is that your code is taking long to process and frames\n are being skipped because of it, It maybe because of some heavy\n processing that you are doing at the heart of your application or DB\n access or any other thing which causes the thread to stop for a while.\n Here is a more detailed explanation:Choreographer lets apps to connect themselves to the vsync, and\n properly time things to improve performance.Android view animations internally uses Choreographer for the same\n purpose: to properly time the animations and possibly improve\n performance.Since Choreographer is told about every vsync events, I can tell if\n one of the Runnables passed along by the Choreographer.post* apis\n doesnt finish in one frame\xe2\x80\x99s time, causing frames to be skipped.In my understanding Choreographer can only detect the frame skipping.\n It has no way of telling why this happens.The message \xe2\x80\x9cThe application may be doing too much work on its main\n thread.\xe2\x80\x9d could be misleading.source : \n Meaning of Choreographer messages in LogcatWhy you should be concernedWhen this message pops up on android\n emulator and the number of frames skipped are fairly small (<100) then\n you can take a safe bet of the emulator being slow \xe2\x80\x93 which happens\n almost all the times. But if the number of frames skipped and large\n and in the order of 300+ then there can be some serious trouble with\n your code. Android devices come in a vast array of hardware unlike ios\n and windows devices. The RAM and CPU varies and if you want a\n reasonable performance and user experience on all the devices then you\n need to fix this thing. When frames are skipped the UI is slow and\n laggy, which is not a desirable user experience.How to fix itFixing this requires identifying nodes where there is or\n possibly can happen long duration of processing. The best way is to do\n all the processing no matter how small or big in a thread separate\n from main UI thread. So be it accessing data form SQLite Database or\n doing some hardcore maths or simply sorting an array \xe2\x80\x93 Do it in a\n different threadNow there is a catch here, You will create a new Thread for doing\n these operations and when you run your application, it will crash\n saying \xe2\x80\x9cOnly the original thread that created a view hierarchy can\n touch its views\xe2\x80\x9c. You need to know this fact that UI in android can be\n changed by the main thread or the UI thread only. Any other thread\n which attempts to do so, fails and crashes with this error. What you\n need to do is create a new Runnable inside runOnUiThread and inside\n this runnable you should do all the operations involving the UI. Find\n an example here.So we have Thread and Runnable for processing data out of main Thread,\n what else? There is AsyncTask in android which enables doing long time\n processes on the UI thread. This is the most useful when you\n applications are data driven or web api driven or use complex UI\xe2\x80\x99s\n like those build using Canvas. The power of AsyncTask is that is\n allows doing things in background and once you are done doing the\n processing, you can simply do the required actions on UI without\n causing any lagging effect. This is possible because the AsyncTask\n derives itself from Activity\xe2\x80\x99s UI thread \xe2\x80\x93 all the operations you do\n on UI via AsyncTask are done is a different thread from the main UI\n thread, No hindrance to user interaction.So this is what you need to know for making smooth android\n applications and as far I know every beginner gets this message on his\n console.I am not writing this to gain more reputation. When you call a PreferenceManager.getSharedPreferences and other similar methods for the first time, the associated .xml file is immediately loaded and parsed in the same thread.One of good ways to combat this issue is triggering first SharedPreference load from the background thread, started as early as possible (e.g. from onCreate of your Application class). This way the preference object may be already constructed by the time you\'d want to use it.Unfortunately, sometimes reading a preference files is necessary during early phases of startup (e.g. in the initial Activity or even Application itself). In such cases it is still possible to avoid stalling UI by using MessageQueue.IdleHandler. Do everything else you need to perform on the main thread, then install the IdleHandler to execute code once your Activity have been fully drawn.
